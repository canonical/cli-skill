# Error Model & Exit Codes

## Overview

Juju uses Go's `github.com/juju/errors` package for error handling, with structured error types and the ability to embed causes. Exit codes follow a simple convention but are not exhaustively documented.

## Exit Code Conventions

| Exit Code | Meaning | When Used |
|-----------|---------|-----------|
| `0` | Success | Command completed without error |
| `1` | General error | Most command failures |
| `2` | Usage / invocation error | Missing arguments, flag errors, invalid syntax, or startup failures (e.g., `cmd.WriteError` in `main.go` returns 2) |

The CLI does not use a rich set of exit codes. Most "expected" failures return 1, while usage/parsing errors return 2. There are no differentiated exit codes for specific error categories (e.g., auth failure, timeout, not found).

## Error Categories

### 1. Usage / Parsing Errors (Exit Code 2)

These are generated by the command framework when arguments or flags are invalid:

```
error: no name specified
error: unrecognized command: juju foo
error: unknown format "xml"
error: a relation must involve two applications
error: --no-wait requires --force
```

Generated by: `cmd.CheckEmpty()`, `gnuflag` parsing failures, `Errorf` in `Init()` methods.

### 2. API / Server Errors (Exit Code 1)

Errors from the Juju controller or cloud provider:

```
error: cannot connect to controller
error: model "foo" not found
error: permission denied
error: machine 0 not found
```

These propagate from the API layer via `errors.Trace()`.

### 3. Network & Transport Errors (Exit Code 1)

Connection failures, timeouts, proxy errors:

```
error: cannot establish the connection
error: dial tcp <addr>: connection refused
```

### 4. Authorization / Login Errors (Exit Code 1)

Authentication and authorization failures:

```
error: permission denied
error: not logged in
```

### 5. Model Migration Errors (Exit Code 1)

Specialized error category with actionable messages:

```
Model "foo" has been migrated to controller "bar".
To access it run 'juju switch bar:foo'.
```

### 6. NotFound / UnrecognizedCommand (Exit Code 1 or 2)

When a subcommand is not recognized:

```
juju: "foo" is not a juju command. See "juju --help".

Did you mean:
        bar
```

Exit code 1 (from the framework); the framework provides fuzzy matching via `FindClosestSubCommand`.

### 7. Configuration Errors

Invalid configuration values:

```
error: invalid config value "xyz" for key "foo": must be a boolean
```

### 8. Feature Flag Errors

When a feature-gated command is used without enabling the flag:

```
error: command "dump-model" requires developer mode
```

Generated by the command registration system.

## Error Message Style

Per DE013 guidance, Juju error messages follow these conventions:

1. **"cannot" not "can't"**: `error: cannot establish the connection` (no contractions)
2. **Passive for missing inputs**: `error: no name specified` (not "You need to provide a name")
3. **Action-oriented for recovery**: `To access it run 'juju switch bar:foo'.`
4. **Precise entity names**: Error messages include model names, controller names, and UUIDs when applicable
5. **Warnings to stderr**: Deprecation warnings, non-fatal advisories go to stderr

### Warnings

Warnings are output via the `ctx.Warningf()` / `cmd.WriteError()` path and go to stderr:

```
warning: "foo bar" is deprecated, use "foo baz" instead
```

### Deprecation Handling

The framework supports `RegisterDeprecated()` for commands and numeric deprecation checks. Deprecated commands still execute but emit warnings. There is no active command deprecation in the current registration, but the framework supports it for future use.

## Per-Command-Group Differences

### Destroy Commands

- `destroy-model --force --no-wait`: Errors during teardown are accumulated and reported at the end
- `destroy-controller`: More detailed error reporting, especially for non-empty models
- `kill-controller`: Designed to work even when the controller is unreachable; different error paths

### SSH Commands

- `ssh`: Connection errors include retry logic with configurable strategy
- `debug-hooks`: Specialized error about tmux state
- `scp`: Uses SCP error codes from the underlying protocol

### run / exec

- Exit code reflects the underlying command's exit code (not Juju's own error)
- Timeout errors: `error: command timed out after 5m0s`
- Background operations: Errors are reported via `show-operation`

### status

- `--retry-count` and `--retry-delay` control retry behavior on API failures
- Connection errors during status are retried, then reported

### Cross-Model Relations (CMR)

- `consume`: Detailed errors about offer URL format, connectivity
- `offer`: Errors about endpoint availability, permissions

## Error Code Stability

Exit codes 0 (success), 1 (error), and 2 (usage) are the only documented codes. There is no formal guarantee about specific exit codes for specific error conditions. Scripts should not rely on exit code differentiation beyond these three values.

## Comparison to DE013 Standard

DE013 recommends that CLI errors should be:

1. **Human-readable** ✅ — Juju uses natural language, includes context (entity names)
2. **Short and succinct** ✅ — Error messages are generally concise
3. **Provide next-step guidance** ⚠️ — Some errors are actionable (migration, login) but many are not
4. **Standard exit codes** ⚠️ — Juju only uses 0/1/2; no differentiated codes for scripting
5. **Use "cannot"** ✅ — Most errors use "cannot" not "can't"
6. **Passive where appropriate** ✅ — Missing arg errors use passive voice
